Q Tell me a prod issue

One production incident I handled was during a high-volume screening run where we started seeing delays and a growing backlog in our processing pipeline.

This was a concern because it directly impacted our SLA for client screening, and downstream systems were starting to get overwhelmed.

My first priority was containment. We slowed down incoming processing and prioritized critical flows so we could stabilize the system and prevent the backlog from growing further.

Then I worked with the team to identify the root cause. We found that under peak load, the system was sending too many requests in parallel to downstream services, which led to timeouts. Those timeouts triggered retries, which further increased load and created a feedback loop.

To fix this, we introduced rate limiting to control how many requests were sent at a time and adjusted our retry strategy so the system wouldn’t amplify the problem under load.

We also improved monitoring so we could detect these patterns earlier.

As a result, we were able to stabilize the system, clear the backlog, and avoid further SLA risk.

The key takeaway for me was that it’s not just about handling high volume, but ensuring the system behaves predictably under load—especially when retries are involved.”

Spring Batch does:

DB → split into partitions → process in parallel → send messages to Kafka
How partitioning works
ClientIdRangePartitioner splits 50M records into chunks:
Partition 1 → clientId 1–10M
Partition 2 → 10M–20M
etc.

Each partition:

runs in parallel thread (Spring Batch worker)
reads DB
publishes messages to Kafka

👉 YES — all partitions publish to the SAME Kafka topic

❗ Important answer to your question

“do they send to same Kafka queue?”

👉 YES.

Same topic
But topic has multiple partitions
2️⃣ Kafka (middle layer)

Kafka topic is configured like:

Topic: screening-events
Partitions: 10 (for example)

When Spring Batch publishes:

messages are distributed across Kafka partitions
(based on key or round-robin)
3️⃣ Kafka Consumers (Processing side)

Now comes real parallel processing

How parallelism happens
# of parallel consumers = # of partitions (max)

Example:

Kafka topic has 10 partitions
You can have up to 10 parallel consumers

Each consumer:

reads from 1 partition
processes messages independently
calls screening service (Clink/RDC)

👉 YES — consumer parallelism comes from Kafka partitions

is your application (microservice)
