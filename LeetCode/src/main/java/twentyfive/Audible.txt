General questions to ask interviewers once a problem is provided:

1) What is the maximum constraint on the input size (N)?
Why you ask: If (N <= 10^3), an (O(N^2)) solution might pass. 
If (N <= 10^6), you are immediately restricted to an (O(N)) or (O(N log N)) algorithm.

2)What is the data range of the values inside the array/stream?
Why you ask: If integers can be negative, standard sliding window approaches or prefix sum assumptions break. If the range is extremely tight (e.g., ages 0–100), you can use an (O(N)) Counting Sort instead of a general (O(N log N)) sort.

Stream/Static

Why you ask: If it is a stream, you cannot call Arrays.sort(). You must maintain state dynamically using structures like a min/max heap, a sliding window deque, or an online algorithm.

4)Are the incoming events or timestamps guaranteed to arrive in chronological order, or do we need to handle out-of-order/late-arriving data?
Why you ask: If data arrives out of order, you must implement a buffering or sorting mechanism before processing, which shifts your space complexity requirements.

Edge Cases and System Failures: This proves you build fault-tolerant production systems.

Why you ask: Shows you handle defensive guardrails from the first line of code.

class LRUCache {

    DLLNode front;
    DLLNode end;
    Map<Integer,DLLNode> lruMap;
    int capacity;
    class DLLNode{
        int key;
        int value;
        DLLNode next;
        DLLNode prev;
        public DLLNode(){};
        public DLLNode(int key, int value){
            this.key=key;
            this.value=value;
        }
    }

    public LRUCache(int capacity) {
        this.capacity=capacity;
        lruMap= new HashMap<>();
        front = new DLLNode();
        end= new DLLNode();
        front.next=end;
        end.prev=front;
    }
    
    public int get(int key) {
        if(lruMap.containsKey(key)){
            DLLNode node = lruMap.get(key);
            bringNodeToFront(node);
            return node.value;
        }
        return -1;
    }
    
    public void put(int key, int value) {
        if(lruMap.containsKey(key)){
            DLLNode node =lruMap.get(key);
            node.value=value;
            bringNodeToFront(node);
        }
        else{
            DLLNode node = new DLLNode(key,value);
            lruMap.put(key,node);
            addNodeToFront(node);
            if(lruMap.size()>capacity){
                evictLRUCache();
            }
        }
    }
    
    //end<- p1<-front
    public void addNodeToFront(DLLNode node){
      DLLNode temp= front.next;
      front.next=node;
      node.prev=front;     
      node.next=temp;
      temp.prev=node;     
    }

    public void bringNodeToFront(DLLNode node){
        removeNode(node);
        addNodeToFront(node);
    }

    public void removeNode(DLLNode node){
        node.prev.next=node.next;
        node.next.prev=node.prev;
    }
    
    public void evictLRUCache(){
        DLLNode tailNode = end.prev;
        removeNode(tailNode);
        lruMap.remove(tailNode.key);

    }
   
}

"To achieve O(1) get and put, I'll combine a HashMap and a Doubly Linked List. The HashMap gives O(1) access to nodes by key. The Doubly Linked List maintains recency order, with the most recently used item near the head and the least recently used near the tail. On every get or update, I'll move the node to the front. When capacity is exceeded, I'll evict the node at the tail."

Why doubly linked list instead of singly linked list?
Because when moving a node to the front, we need O(1) removal.

With a singly linked list:

remove(current) requires finding the previous node first.
That becomes O(n).
Doubly linked list gives:

node.prev.next = node.next;
node.next.prev = node.prev;

in O(1).

Why use dummy head and tail nodes?
	Dummy nodes eliminate edge cases for inserting and deleting at the boundaries. Every real node always has both a previous and next node, simplifying the logic and reducing bugs.


The current implementation is not thread-safe because both the HashMap and doubly linked list can be modified concurrently, causing corruption or inconsistent state. The simplest fix is synchronizing get and put operations so modifications to DLL and map become atomic. A better approach is using a ReentrantLock to protect all modifications to both data structures. ConcurrentHashMap alone is insufficient because the linked list updates are still unsafe.


ReentrantLock is superior to the synchronized keyword because it gives you fine-grained programmatic control over thread orchestration.While synchronized is an "all-or-nothing" structural block built into the Java language, ReentrantLock is a flexible API class that allows you to avoid thread starvation, prevent deadlocks, and handle heavy traffic pipelines gracefully

        //"I intentionally modeled the edge from a service to its dependency. Since DFS adds nodes in post-order, a service is added only after all of its dependencies have been added. That naturally produces a valid boot sequence without needing a final reverse."

class Solution {
    //keep max at front, remove from back...
    /*Why remove from the back?
    /*I would say:The front of my deque always contains the largest element in the current window. The back contains smaller candidate elements. When a larger element arrives, I remove smaller elements from the back because they can never become the maximum in any future window. When an element leaves the sliding window, I remove it from the front if its index is no longer valid. Therefore the front of the deque always contains the maximum for the current window.
    That explanation is 100% correct and matches your natural visualization:
    back <-- front
    small <-- large*/
	//This is a monotonic decreasing deque
    /*A single call can be O(N) in the worst case if many expired timestamps are removed at once. However, each timestamp enters the deque once and leaves once, so across N events there are at most 2N deque operations. Therefore the amortized complexity of shouldProcessEvent() is O(1).*/
    public int[] maxSlidingWindow(int[] nums, int k) {
        int n = nums.length;
        int[] result= new int[n-k+1];
        Deque<Integer> dq= new ArrayDeque<Integer>();
        for(int i=0;i<n;i++){
            //4,3,2    
            while(!dq.isEmpty() && dq.peekFirst()<=i-k)
                  dq.pollFirst();
            
            //4,3,2,1
            while(!dq.isEmpty() && nums[i]>nums[dq.peekLast()]){
                dq.pollLast();
            }
            dq.offerLast(i);
            //
            if(i>=k-1){
                result[i-k+1]=nums[dq.peekFirst()];
            }
                  
        }
        return result;

    }
}

class Solution {
    public int[] dailyTemperatures(int[] temperatures) {
        int n = temperatures.length;
        int[] result = new int[n];
        Stack<Integer> st = new Stack<>();

        for(int j = n - 1; j >= 0; j--) {

            while(!st.isEmpty() &&
                  temperatures[j] >= temperatures[st.peek()]) {
                st.pop();
            }

            if(!st.isEmpty()) {
                result[j] = st.peek() - j;
            }

            st.push(j);
        }
        return result;
    }
