Q2: Tell me about your current architecture
============================================

"At a high level, the architecture sits between upstream business systems and downstream vendor screening platforms.

Ingestion Layer: Upstream systems — onboarding, client lifecycle, account updates — send client and account data to our platform for screening.

Orchestration Layer: Our platform exposes APIs with well-defined, vendor-agnostic data contracts that standardize how screening requests are submitted and how results are interpreted. Based on the use case, we route to either a real-time flow (e.g., onboarding) or a batch flow (periodic re-screening).

Batch Processing Layer: For large-volume batch runs, we use Spring Batch with a ClientIdRangePartitioner that splits tens of millions of records into ID-range partitions. Each partition runs in a parallel worker thread, reads from the database, and publishes messages to a shared Kafka topic. This gives us producer-side parallelism.

Event Layer (Kafka): The Kafka topic itself is partitioned (e.g., 10 partitions), providing the second layer of parallelism. Kafka consumers process messages independently — each consumer reads from one partition, invokes downstream vendor APIs (Clink or RDC), and handles results.

Resilience: Consumers are idempotent, with retry mechanisms (exponential backoff + jitter) and dead-letter queues (DLQ) for transient failures. We also apply rate limiting and circuit breakers to protect downstream vendor systems.

Normalization & Alerting: Vendor responses are normalized into a canonical format, alerts are generated in Actimize, and reconciliation processes ensure what we received from vendors matches what we expect over time.

Observability: Prometheus and Grafana provide real-time monitoring of throughput, latency, error rates, and backlog trends.

Key Design Principle: The core platform is generic — variability is pushed to the edges. APIs return consistent data; consuming systems apply their own filtering or business logic. This avoids tight coupling and allows the system to scale across multiple lines of business.

Two layers of parallelism (important to articulate):
1) Spring Batch = producer-side parallelism (DB reading + Kafka publishing)
2) Kafka consumers = processing-side parallelism (vendor API calls)

Spring Batch workers do NOT call vendors directly. They only publish to Kafka. Each Kafka consumer is a microservice instance that calls Clink/RDC independently."

Layer 2 — Queryable Reasoning Agent:

Architecture:
It followed a RAG-style pattern — LLM-based semantic extraction first, then an agent layer for contextual reasoning and Q&A, with validation against source SQL to minimize hallucination. We implemented guardrails including source citation and confidence scoring.

Impact:
- Engineer onboarding time for population logic reduced from ~6 weeks to ~2 weeks
- Compliance Q&A cycles reduced by roughly 60%
- Turned deeply embedded, opaque logic into something transparent and queryable
- Both engineering and compliance teams could now reason about screening behavior independently

Key Lesson:
Systems don't just need to scale technically — knowledge needs to scale as well. Since then, I've been proactive about making system behavior explicit from the start."

Q5: Tell me a time you faced conflict / influenced decisions
============================================================= 

Story 1 — Influencing POC Scope (Vendor Integration)
-----------------------------------------------------

"During a vendor integration, there was a push from the business side to manually extract data for ~20 million clients to support a POC.

The ask came quickly, and the expectation was to proceed. But from an engineering and operational perspective, it introduced significant risk in scalability and data handling — we estimated roughly 3 weeks of manual effort, plus ongoing data governance concerns.

Instead of pushing back directly, I reframed the discussion. I broke down what we were actually trying to validate with the POC and quantified the scale and risks of the proposed approach. I then proposed an alternative — a smaller, representative dataset of ~50K records already available — that would validate the same hypotheses without the risk.

I also clarified the difference between what's acceptable for a POC versus a production-ready solution, aligning expectations early.

This shifted the conversation from positions to trade-offs. The team aligned on the safer approach, we avoided ~3 weeks of unnecessary effort, and still achieved the POC objective within the original timeline.

Takeaway: Influencing decisions is less about saying no and more about making assumptions and trade-offs explicit so everyone can make informed choices."

Story 2 — Influencing API / Data Contract Design (EDD)
-------------------------------------------------------

"I designed APIs and data contracts for an Enhanced Due Diligence workflow focused on real-time screening.

During design, there was a push to tailor APIs for specific consumer use cases — embedding filtering and business logic directly into the API layer. While that would have worked short-term, I recognized it would tightly couple the API to individual consumers and make it harder to scale.

I proposed keeping the API generic and pushing filtering/interpretation logic to the edges. I explained the trade-offs clearly: embedding logic reduces initial consumer effort but increases long-term complexity, reduces reusability, and makes cross-team changes riskier. A generic API requires slightly more upfront effort but yields a far more scalable and maintainable system.

The team aligned. The API now serves 4 different consuming systems without any core modifications — each applies its own business logic on the response.

Takeaway: Design clean boundaries — keep core systems stable and push variability to the edges."

Story 3 — Influencing Real-time vs Batch Design
-------------------------------------------------

"During pipeline design, there was a push to make most flows real-time to improve responsiveness.

I recognized that pushing everything real-time would increase complexity, operational risk, and infrastructure cost without clear value for all scenarios.

I broke down the use cases and evaluated where real-time genuinely added value (high-risk onboarding — ~5% of total volume) versus where batch was sufficient (periodic re-screening — ~95% of volume). I presented the trade-off: responsiveness vs. system complexity, failure handling overhead, and infrastructure cost.

I proposed a hybrid approach — real-time only for critical paths, batch for the rest. This reduced the real-time infrastructure footprint by roughly 80% compared to the original all-real-time proposal.

Takeaway: Influence decisions by grounding them in system behavior, data, and trade-offs — not preferences."

Q9: How is your approach to system design?
===========================================

"My approach to system design is constraint-first and iterative. Here's how I think about it:

1. Start with constraints, not solutions.
I first understand the regulatory, scale, and auditability requirements. In financial crimes tech, every design decision must balance performance with the ability to trace and explain outcomes. For example, before designing our batch pipeline, I mapped out SLA requirements (24-hour processing window for 50M records), compliance needs (full audit trail), and failure tolerance expectations.

2. Define clear boundaries and contracts.
I design canonical, vendor-agnostic data contracts and clean API boundaries. The core platform remains generic and stable; variability is pushed to the edges. This is what allowed our EDD APIs to serve 4 different consumers without core changes.

3. Separate concerns by flow type.
I evaluate which flows need to be real-time (e.g., high-risk onboarding — ~5% of volume) vs. batch (periodic re-screening — ~95%), applying the right pattern to each and avoiding unnecessary complexity.

4. Design for parallelism and decoupling.
I use partitioning (Spring Batch with ClientIdRangePartitioner) for producer-side parallelism and Kafka for event-driven distribution and consumer-side parallelism. This gives two independent scaling dimensions — we can scale data extraction and processing separately.

5. Build resilience into the architecture.
Idempotent consumers, retry with exponential backoff and jitter, dead-letter queues, rate limiting, circuit breakers, and reconciliation processes — all designed so the system behaves predictably under load and failure.

6. Embed observability from day one.
Prometheus, Grafana, and structured logging for throughput, latency, error trends, and SLA compliance monitoring. We set up alerts for backlog growth rate and p99 latency thresholds before going to production.

7. Validate, don't assume.
Especially with external vendors, I take a validation-driven approach — controlled tests, targeted demos, and incremental iterations rather than locking the architecture based on assumptions.

The key principle that ties it all together: design for what you know, build safeguards for what you don't, and keep the core stable so the system can evolve without constant rewrites."

Q10: Tell me a prod issue
===========================

"During a high-volume screening run, we started seeing delays and a growing backlog in our processing pipeline. Within about 2 hours, the backlog grew to over 400K unprocessed messages, and p99 latency spiked from ~200ms to over 12 seconds. This directly impacted our SLA for client screening, and downstream systems were getting overwhelmed.

Containment (Immediate):
I slowed incoming processing by reducing the Spring Batch partition count and prioritized critical flows (high-risk onboarding) so we could stabilize the system and prevent the backlog from growing further.

Root Cause Analysis:
Under peak load, the system was sending too many parallel requests to downstream vendor services, causing timeouts. Those timeouts triggered retries, which further increased load — creating a feedback loop (retry storm). Essentially, our retry strategy was amplifying the problem instead of recovering from it.

Fix:
- Introduced rate limiting to control concurrent outbound requests (capped at 50 concurrent calls per consumer instance)
- Adjusted the retry strategy — exponential backoff with jitter and a max-retry cap of 3 — so the system wouldn't amplify problems under load
- Added circuit breakers (half-open state with health probes) to fail fast when downstream services were degraded
- Improved monitoring and alerting — added Prometheus metrics for retry rate, backlog growth rate, and downstream response time, with Grafana alerts that trigger before SLA risk

Result:
Stabilized the system within 45 minutes of containment, cleared the full backlog within 4 hours, and avoided SLA breach. The retry storm pattern has not recurred since.

Takeaway: It's not just about handling high volume — it's about ensuring the system behaves predictably under load. Retries without backpressure can turn a minor downstream hiccup into a cascading failure. Now, resilience patterns (rate limiting, circuit breakers, capped retries) are part of our standard service template."

Q12: How did you scale the system?
====================================

"In our Financial Crimes Technology group, we screen ~20 million client records monthly for sanctions, PEP, and adverse media. Our legacy batch platform was slow, prone to timeouts, and risked breaching compliance SLAs during onboarding spikes. The goal was to redesign for 50+ million records with strict 24-hour processing SLAs.

I approached scale at multiple layers:

1. Separated real-time and batch flows.
Real-time only for latency-sensitive scenarios like onboarding (~5% of volume); batch for large-volume periodic screening (~95%). This avoided unnecessary real-time complexity and reduced infrastructure cost.

2. Producer-side parallelism (Spring Batch).
Partitioned the dataset using ClientIdRangePartitioner (client ID ranges). For 50M records with gridSize=5, each partition handles ~10M records in parallel. Each worker thread reads from the database and publishes to Kafka.

3. Decoupled via Kafka.
All Spring Batch workers publish to the SAME Kafka topic (e.g., screening-events), which itself has multiple partitions (e.g., 10). This decouples data extraction from screening processing, allowing each side to scale independently.

4. Consumer-side parallelism (Kafka consumers).
Parallel consumers — one per Kafka partition — process messages independently and call downstream vendor APIs (Clink/RDC). Consumer count scales with Kafka partition count. With 10 partitions, we run 10 parallel consumers.

5. Resilience at scale.
- Idempotent consumers for safe retries
- Exponential backoff with jitter + max 3 retries
- Dead-letter queues for poison messages and transient failures
- Rate limiting (50 concurrent calls per consumer) and circuit breakers to protect downstream systems
- Spring Batch checkpointing for failure recovery (resume from last successful chunk)

6. Stable, generic APIs.
Core APIs and data contracts remain generic; variability pushed to the edges. As consumers and use cases grew, the core system required zero modifications.

7. Observability.
Prometheus and Grafana for real-time monitoring of throughput, latency, error trends, retry rates, and SLA compliance. Alerts on backlog growth rate and p99 thresholds.

Results:
- Processing time reduced by ~40%
- Alert backlogs eliminated
- Scaled from 20M to 50M+ records reliably
- Zero SLA breaches post-redesign
- Platform became the backbone for global client screening across Wealth and ISG

Key insight: The goal wasn't just scaling throughput — it was scaling safely. Every design decision balanced performance with auditability and correctness in a regulated environment."
