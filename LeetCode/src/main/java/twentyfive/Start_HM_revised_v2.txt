Q2: Tell me about your current architecture
============================================

1. Business Problem

I built a real-time client screening platform that evaluates customers against sanctions, PEP, and adverse media datasets. The platform integrates with external providers such as RDC and Bridger through their APIs, which perform fuzzy matching of customer information against compliance watchlists and adverse media sources.

The key challenge here was balancing low-latency decisioning—especially for onboarding flows—with high throughput and strict auditability requirements.

2. Core Architecture

To address this, we designed the system to be API-first for real-time screening, while using Kafka for asynchronous processing such as audit logging and alerting.”

Upstream systems call our API Gateway and submit screening requests using a vendor-agnostic data model that I designed. The screening service validates the request and transforms it into the format required by the underlying screening provider, such as Bridger or RDC. We then call the Bridger API, which performs fuzzy client name+ address+ matching against compliance lists like OFAC and PEP.

Once we receive the vendor response, we normalize the vendor output into canonical internal response model, perform decision mapping to determine outcomes such as hit, no-hit, or review.The decision is returned synchronously to the calling application to support real-time use cases like onboarding. (200 - 500 mS SLA)

	
3. Kafka Distribution

In parallel, we publish the normalized screening result to Kafka. Kafka acts as our distribution layer for audit storage, Actimize alerting, and downstream consumers.

We keep Kafka out of the critical decisioning path to maintain low latency. The onboarding application only requires the screening decision, while audit persistence and downstream integrations can be processed asynchronously.

This is a very strong tradeoff discussion.

4. Event-Driven Ingestion

In addition to synchronous APIs, we also support event-driven ingestion through Kafka for high-throughput scenarios.

In this model, requests are received from a Kafka topic rather than through the API Gateway. The same screening service processes the request, performs validation and transformation, invokes the external screening provider, and applies the same decisioning logic.

Because these workloads are not latency sensitive, we treat the flow as asynchronous. Instead of returning a synchronous response to the caller, the screening result is published to a Kafka topic.

Downstream systems can consume the event directly, and if the originating client needs the result, it can subscribe to the topic.( use a webhook callback, or poll using a requestId)

The key tradeoff is that the API path optimizes for low-latency decisioning, while the Kafka ingestion path optimizes for throughput, scalability, and loose coupling. For onboarding and other real-time use cases, we use the API path. Kafka ingestion is used when latency is less critical and throughput is high.

5. Use Cases

We support three primary patterns:

Different consumers have different requirements. Some require immediate decisions, while others prioritize throughput and asynchronous processing. Supporting both allows us to reuse the same core screening service while providing the appropriate integration pattern for each use case.

Real time Use case: Customer onboarding as part of enhanced due diligence (For institutional clients)


Notice how the use cases come after the architecture.

6. Scaling

The screening service scales horizontally, increasing both API handling capacity and Kafka consumer throughput.

Kafka parallelism is bounded by partition count, while overall throughput is ultimately constrained by external screening providers. To protect downstream systems, we implement controlled concurrency, rate limiting, and backpressure mechanisms.

7. Backpressure

If asked:

What do you mean by backpressure?

Answer:

Kafka consumers pull messages rather than having messages pushed to them. When downstream systems become overloaded, consumers can intentionally slow consumption rates, preventing retry storms, resource exhaustion, and cascading failures across the platform.

Finally, we separated decisioning from downstream processing. Once a screening decision is made, results are published to Kafka for audit storage, alerting, and other integrations, keeping those workloads off the critical path.

Q9: How is your approach to system design?
===========================================

"My approach to system design is constraint-first and iterative. Here's how I think about it:

First, I start with constraints rather than solutions. Before discussing technology choices, I want to understand the business requirements, scale expectations, latency targets, regulatory requirements, and failure tolerance. Those constraints usually drive most architectural decisions.

Second, I focus on defining clear boundaries and contracts. I prefer stable, canonical APIs and data models that isolate consumers from implementation details. This reduces coupling and makes systems easier to evolve over time.

Third, I separate concerns and choose the right processing model for the problem. Not every workflow needs to be real-time. I evaluate latency requirements, throughput needs, and operational complexity, then choose the appropriate architecture, whether that's synchronous APIs, event-driven processing, or batch workloads.

Fourth, I design for resilience and observability from the beginning rather than treating them as afterthoughts. That includes retries, idempotency, rate limiting, circuit breakers, logging, monitoring, and auditability so the system behaves predictably under both normal and failure conditions.

Finally, I validate high-risk assumptions early, especially when external dependencies are involved. Rather than locking a design based on incomplete information, I prefer targeted testing, prototypes, and incremental iterations to reduce risk before committing to major architectural decisions.

Q10: Tell me a prod issue
===========================


One production issue I investigated involved our Wealth Management event-driven screening pipeline.

We started seeing Kafka consumer lag grow steadily and end-to-end processing times increase for screening events. Since this pipeline supports high-volume screening workflows, sustained lag growth was a concern because it meant the system was processing events more slowly than they were being produced.

My first step was determining where the bottleneck existed. I looked at Kafka consumer lag, consumer throughput, application logs, JVM health metrics, and downstream response times from our on-prem Bridger screening platform.

Initially, it wasn't clear whether the issue was Kafka, the consumer application, or Bridger itself.

I started by validating the health of the consumers. CPU utilization was normal, heap usage was stable, GC activity looked healthy, and there was no evidence of thread contention or resource exhaustion. That helped rule out the consumer application as the primary bottleneck.

Next, I correlated consumer throughput with Bridger response times. That's where I found the issue. Response times from Bridger had increased significantly. Because each consumer thread spent longer waiting for screening responses, overall throughput dropped and messages began accumulating in Kafka.

The key realization was that the growing consumer lag was a symptom rather than the root cause. The actual bottleneck was downstream processing latency in the screening platform.

I worked with the Bridger support and infrastructure teams to investigate the issue. Together we identified resource saturation in the environment, which was causing requests to queue and response times to increase under load. The team adjusted capacity and configuration on the Bridger side, and we closely monitored lag growth, throughput, and response-time metrics throughout the process.

Once response times returned to normal, consumer throughput recovered and the backlog cleared naturally.

After the incident, we enhanced our monitoring by adding dashboards and alerts around Kafka lag, consumer throughput, and Bridger response times so we could detect similar degradation much earlier.

The biggest lesson for me was that in distributed systems it's important to identify the true bottleneck before taking corrective action. Looking only at Kafka lag could have led us to simply add more consumers, but that would have increased pressure on an already constrained downstream system without improving throughput. Correlating lag, throughput, JVM metrics, and dependency metrics allowed us to identify the real issue much more quickly.

Instead of figuring out LexID behavior alone, I brought two mid-level engineers in. 
I walked them through the first round of structured test scenarios against the SaaS instance — showing how to validate vendor behavior against our assumptions. 
Then I handed them ownership of the next batch, including edge cases like partial matches and entity merges.

I also had them own the LexID-to-canonical-alert mapping design. I gave constraints, they proposed, I challenged, they iterated.

Within 6 weeks, they went from no LexID exposure to independently owning entity resolution for the full integration.
 One of them authored the LexID reference document that compliance now uses.

POCs are one of the best training opportunities — contained scope, real stakes, hands-on learning."

Q14: Tell me about a failure or a time something didn't go as expected
========================================================================
When I joined the financial crimes screening team, most of the team, including the lead, was based in India. 
I expected a structured ramp-up with code walkthroughs, knowledge transfer sessions on screening logic and business processes, someone showing me how the system worked.

That didn't happen. The team was busy, time zones made real-time collaboration hard, and there wasn't a formal onboarding path. 
Looking back, the mistake I made was waiting too long for that support structure to materialize.

Once I recognized it wasn't going to happen on its own, I shifted my approach. I started reverse-engineering the codebase — tracing flows end-to-end, 
reading the legacy SQL population logic, and setting up targeted async conversations instead of waiting for broad walkthroughs. 
I documented what I learned along the way.

That deep dive uncovered real issues that had gone unnoticed. I found inconsistencies in the client scoping logic where inclusion and exclusion rules weren't being applied consistently, which resulted in over-screening certain clients and generating unnecessary alerts.

Within a couple of months, I had deep system understanding — arguably deeper than passive onboarding would have given me,
because I had to reason through why the system worked the way it did rather than simply being shown how it worked.

The lesson: don't wait for context — go get it. That experience shaped how I operate now. 
It's why I drive clarity proactively, why I led the AI-assisted documentation initiative to make screening logic queryable, and why I structure onboarding 
deliberately for engineers joining my team — like the LexID training during the Bridger POC."

BATCH

"At a high level, the architecture sits between upstream business systems and downstream vendor screening platforms.

Ingestion Layer: Upstream systems — onboarding, client lifecycle, account updates — send client and account data to our platform for screening.

Orchestration Layer: Our platform exposes APIs with well-defined, vendor-agnostic data contracts that standardize how screening requests are 
submitted and how results are interpreted. Based on the use case, we route to either a real-time flow (e.g., onboarding) or a batch flow (periodic re-screening).

Batch Processing Layer: For large-volume batch runs, we use Spring Batch with a ClientIdRangePartitioner 
that splits tens of millions of records into ID-range partitions. Each partition runs in a parallel worker thread, 
reads from the database, and publishes messages to a shared Kafka topic. This gives us producer-side parallelism.

Event Layer (Kafka): The Kafka topic itself is partitioned (e.g., 10 partitions), providing the second layer of parallelism.
 Kafka consumers process messages independently — each consumer reads from one partition, 
 invokes downstream vendor APIs (Clink or RDC), and handles results.

Resilience: Consumers are idempotent, with retry mechanisms (exponential backoff + jitter) 
and dead-letter queues (DLQ) for transient failures. We also apply rate limiting and circuit breakers to protect downstream vendor systems.

Normalization & Alerting: Vendor responses are normalized into a canonical format, alerts are generated in Actimize, 
and reconciliation processes ensure what we received from vendors matches what we expect over time.

Observability: Prometheus and Grafana provide real-time monitoring of throughput, latency, error rates, and backlog trends.

Key Design Principle: The core platform is generic — variability is pushed to the edges.
 APIs return consistent data; consuming systems apply their own filtering or business logic. This avoids tight coupling and
 allows the system to scale across multiple lines of business.

Two layers of parallelism (important to articulate):
1) Spring Batch = producer-side parallelism (DB reading + Kafka publishing)
2) Kafka consumers = processing-side parallelism

Spring Batch workers do NOT call vendors directly. They only publish to Kafka. Each Kafka consumer is a microservice instance that calls Clink/RDC independently."
