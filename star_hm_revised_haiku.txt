Q2: TELL ME ABOUT YOUR ARCHITECTURE (75 seconds)
───────────────────────────────────────────────────────────────────────────────
The architecture sits between upstream business systems and downstream vendor 
screening platforms—it's an orchestration layer.

Upstream systems (onboarding, client lifecycle) send data to us. We expose APIs 
with well-defined contracts that standardize requests and normalize vendor 
responses. This allows consuming systems to interact with a consistent interface 
regardless of which vendor (Clink for sanctions, RDC for adverse media) is 
underlying.

We split processing into real-time (onboarding) and batch (periodic screening), 
then integrate with vendors who return matches or alerts. A key piece is how we 
normalize those responses consistently, handle scenarios like duplicate matches 
and partial failures, and ensure reconciliation aligns what we receive with 
expectations.

The design keeps the core platform generic and pushes variability to the edges—
APIs return consistent data; consuming systems apply their own business logic. 
This avoids tight coupling and scales across multiple use cases.

Q4: TELL ME A TIME YOU IMPROVED A PROCESS / USED AI / DEALT WITH LEGACY SYSTEMS
───────────────────────────────────────────────────────────────────────────────
One of my biggest wins was turning tribal knowledge about screening behavior 
into an explicit, queryable system.

**The Problem:** Complex legacy SQL-based screening population logic was embedded 
in database objects, undocumented, and understood by only a few people. As the 
system grew, we saw repeated questions, inconsistent interpretations, and 
slower decision-making.

**The Solution:** I built an AI-assisted two-layer system:
  - **Layer 1:** Used Claude Sonnet to analyze legacy SQL and extract structured 
    representations of table relationships, inclusion/exclusion rules, and business 
    logic (e.g., 401k retirement filters, role-based eligibility).
  - **Layer 2:** Implemented a GPT-based agent on top that allows stakeholders 
to ask conversational questions like "Why are 401k accounts excluded?" or 
"Which roles are in sanctions screening?"

**Result:** Turned deeply embedded logic into something transparent and queryable. 
Reduced tribal knowledge dependency, improved onboarding, and gave engineering 
and compliance teams a shared way to reason about screening behavior.

**Key Lesson:** Systems don't just need to scale technically—knowledge needs to 
scale. Since then, I'm proactive about making behavior explicit from the start.

Q5: TELL ME A TIME YOU FACED CONFLICT / INFLUENCED DECISIONS (90 seconds)
───────────────────────────────────────────────────────────────────────────────
STORY 1 — Vendor Integration POC (20M Client Manual Extract)

**Situation:** Business pushed to manually extract 20M client records to support 
a POC. The ask was urgent; the expectation was to proceed.

**Problem:** From engineering perspective, this introduced massive scalability 
and data-handling risk with no clear ROI.

**Approach:** Instead of saying "no," I reframed it. I broke down what we were 
actually validating and quantified risks. Then I proposed: use a smaller, 
representative dataset we already had. Same validation, zero manual work, lower 
risk.

**Outcome:** Team aligned on the safer approach. We validated the integration 
without unnecessary effort and maintained long-term viability.

**Key Insight:** Influencing is about making trade-offs explicit, not saying no.

───────────────────────────────────────────────────────────────────────────────

STORY 2 — API Design (Enhanced Due Diligence Workflow)

**Situation:** Pressure to tailor APIs for specific consumer use cases, embedding 
filtering and business logic directly in the API layer.

**Problem:** Short-term convenience → long-term tight coupling. New use cases 
would require constant API changes.

**Approach:** I proposed keeping the API generic and pushing filtering to 
consumer edges. Core service stays stable; consuming systems apply their own logic.

**Trade-off Explained:** Embedding logic reduces initial consumer effort but 
increases long-term complexity, reduces reusability, and makes changes riskier. 
Generic APIs require slightly more upfront effort but give us a scalable, 
maintainable system.

**Outcome:** Team aligned. API scaled across consumers without becoming tightly 
coupled to any one use case.

**Key Insight:** Design clean boundaries—keep core systems stable, push 
variability to edges.

───────────────────────────────────────────────────────────────────────────────

STORY 3 — Real-Time vs. Batch Trade-Off

**Situation:** Push to make most flows real-time to improve responsiveness.

**Problem:** Everything real-time = complexity, operational risk, cost without 
clear value for all use cases.

**Approach:** I broke down scenarios and evaluated where real-time actually 
added value (high-risk onboarding) vs. where batch was sufficient. Presented 
as responsiveness vs. complexity trade-off.

**Solution:** Hybrid—real-time only for critical paths, batch for the rest.

**Outcome:** More balanced design, avoided unnecessary complexity, met business 
needs.

**Key Insight:** Ground decisions in system behavior and trade-offs, not 
preferences.

Q9: HOW IS YOUR APPROACH TO SYSTEM DESIGN? (90 seconds)
───────────────────────────────────────────────────────────────────────────────
I design for three pillars:

**Decoupling & Generic Core:** Keep core platforms generic; push variability to 
edges. This lets systems scale across consumers without constant modification.

**Validation-Driven Under Uncertainty:** When you don't have full information, 
break problems into pieces, validate critical behaviors early through controlled 
testing and incremental iterations. Don't lock architecture upfront on 
assumptions.

**Auditability Alongside Performance:** In regulated systems, every design 
decision must balance throughput with the ability to trace and explain outcomes. 
This means reconciliation processes, audit controls, idempotent consumers, 
dead-letter queues, rate limiting—all designed in upfront.

**Real Example:** The Bridger migration. Instead of waiting for full prod access 
(which we didn't have), we used SaaS demos to validate key behaviors, built 
safeguards, and designed a hybrid real-time/batch split that reduced complexity 
during onboarding uncertainty.

Design is always a negotiation between competing constraints. The goal is making 
those trade-offs explicit and building systems that don't break when 
constraints shift.

Q10: TELL ME A PROD ISSUE (75 seconds)
───────────────────────────────────────────────────────────────────────────────
High-volume screening run. Processing pipeline backlog grew, directly impacting 
SLA. Downstream systems were overwhelmed.

**Immediate Actions:**
- Contained: slowed ingestion, prioritized critical flows
- Root Cause: Under peak load, system sent too many parallel requests to 
vendors → timeouts → retries → feedback loop

**Fix:**
- Rate limiting to control parallelism
- Adjusted retry strategy so failures didn't amplify under load
- Improved monitoring for early detection

**Result:** Stabilized system, cleared backlog, avoided SLA risk.

**Key Takeaway:** It's not just about handling volume—it's ensuring the system 
behaves predictably under load, especially with retries involved. Design 
protection mechanisms (rate limiting, circuit breakers) upfront.

Q12: HOW DID YOU SCALE THE SYSTEM? (Technical Deep Dive)
───────────────────────────────────────────────────────────────────────────────
Scaled from 20M to 50+ million records/month while reducing processing time 40%.

**Multi-Layer Approach:**

1. **Separate Real-Time & Batch:** Real-time only for latency-sensitive 
   scenarios (onboarding). Batch for high-volume periodic screening. Avoided 
   unnecessary real-time complexity.

2. **Asynchronous Design:** Decoupled ingestion from downstream processing so 
   components scale independently. Used Kafka for event-driven communication—
   handle spikes more effectively, process workloads resilient and scalable.

3. **Spring Batch + Partitioning:** Client ID range partitioning allowed parallel 
   processing across workers. Partitions published to same Kafka topic; parallel 
   consumers read from topic partitions independently.

4. **Generic APIs, Variable Edges:** Core APIs and data contracts stable and 
   generic; consumers apply their own filtering/logic. Ensured no constant core 
   modifications as use cases grew.

5. **Reliability at Scale:**
   - Idempotent consumers (handle retries safely)
   - Retry mechanisms + dead-letter queues (vendor timeouts, database locks)
   - Rate limiting + circuit breakers (protect downstream)
   - Reconciliation processes (ensure consistency over time)

6. **Observability:** Prometheus + Grafana monitoring throughput, latency, error 
   trends in real time.

**Results:** 40% processing time reduction. Eliminated alert backlogs. 50M 
record reliability. Consistent SLA compliance.

**Core Principle:** Scale throughput AND scale safely. In regulated systems, 
every decision balances performance with auditability.

and why I structure onboarding deliberately for engineers on my team.

KEY SYSTEM DESIGN PRINCIPLES
───────────────────────────────────────────────────────────────────────────────
1. DECOUPLE VENDORS FROM CONSUMERS
   → Vendor-agnostic APIs
   → Normalize responses consistently
   → Changes to downstream vendors don't impact upstream

2. PUSH VARIABILITY TO THE EDGES
   → Core platform = stable, generic
   → Consumers apply their own business logic
   → Reduces need for core modifications as use cases grow

3. VALIDATION-DRIVEN UNDER UNCERTAINTY
   → Don't lock architecture on assumptions
   → Validate critical behaviors early
   → Incremental iterations as reality becomes clearer
   → Build safeguards (reconciliation, audit controls)

4. DESIGN FOR RELIABILITY AT SCALE
   → Async + event-driven (Kafka for decoupling)
   → Idempotent consumers (safe retries)
   → Rate limiting + circuit breakers (protect downstream)
   → DLQs for transient failures
   → Reconciliation ensures consistency

5. BALANCE PERFORMANCE WITH AUDITABILITY
   → In regulated systems, traceability matters as much as throughput
   → Every design decision: Can we explain what happened?
   → Monitor + alert on patterns before they impact SLA
